Submit A Story

If you would like to submit a story:

  1. Check our editorial policy for what we cover - http://api.report/editorial-policy.html
  2. You can submit in two ways:
    1. (preferred) as a pull request) - see below.
    2. by email.
  3. You can also become a regular contributor - see below.

All post are subject to editorial control and we operate on a strictly best effort basis. We reserve the right to reject any post without giving reasons.

Become A Contributor

The API Report is open for anyone in the API comunity to post, all you need is a Github account. The API Report runs on Github, using Jekyll and Github Pages as the front-end site you experience. To submit a post, all you to is fork the repository, and add your story under the _posts folder. If you need a primer on how to craft a Jekyll post, visit the Jekyll website, as they provide some great content to get you up to speed.

When ready, submit a pull request for your story. Once a pull request is received on the gh-pages branch of API Report, an editor will make a decision on quality, and whether it meets the format guidelines below. Not all posts will be accepted, but rejections will come with feedback on how to change and make more acceptable.

Story Format

The API Report is meant to provide an easy to read format, allowing anyone to quickly scan the daily API news, and if they want more detail they can visit linked websites for more information.

After four years of publishing API analysis on API Evangelist, you start to get a feel for the format readers prefer, something I've adapted to be the API Report format:

  • Informative, catchy title
  • Simple opening paragraph
  • Details of the news in bullets or paragraph format
  • Quick wrapup paragraph
  • Single photo
  • Link to more information

The goal is to provide a summary of the news, without adding much analysis or opinion, allowing readers to quickly get up to speed and decide if they want to click, and receive more information.

While the editorial process is fairly open, we will be looking for posts that stick to this simple format, avoiding longer pieces that are over 500 words. We will provide feedback with any posts that are rejected, giving authors a chance to better understand the format.

If you have any feedback, submit as issue, or fork the repository and make the changes, and submit a pull request.

Posting Checklist

Here are a few things to consider when submitting your post:

  • Make your post file name in this format YYYY-MM-DD-title-of-your-post.html
  • Make sure your YAML header material uses quotes for text or single quotes when appropriate
  • Put all your photos under /image/[github user folder] - which you will have to create the first time

More items will be added to the checklist as the project matures, so check back often. If you have any problems posting submit a Github issue for the repository.
